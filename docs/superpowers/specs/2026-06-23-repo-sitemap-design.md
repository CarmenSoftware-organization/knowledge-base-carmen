# Repo Sitemap — Design Spec

**Date:** 2026-06-23
**Status:** Approved (design)
**Topic:** A `sitemap.md` that maps the whole monorepo, for human onboarding and AI/Claude context.

## Problem

The repo is a monorepo (`backend/` Go API, `frontend/` Next.js, `contents/` markdown source-of-truth, plus `scripts/`, `docs/`, `manual/`, root config). A newcomer — or Claude/AI navigating the repo — has no single map that answers "what is each part, where does it live, and where do I look next." `CLAUDE.md` gives instructions and `README.md` gives setup, but neither is a structural map.

## Goal

Create a single `sitemap.md` at the repo root that:

- Serves **human onboarding** — a newcomer reads it and understands the shape of the project.
- Serves as **AI/Claude context** — discoverable at root, structured enough to navigate from.
- Complements `CLAUDE.md` (instructions) and `README.md` (how to run) without duplicating them.

### Non-goals

- Not an SEO `sitemap.xml`.
- Not a per-file listing of all 250+ markdown files under `contents/`.
- No git hook / CI enforcement in this iteration (a `--check` mode is provided for later opt-in).

## Approach

**Hybrid:** hand-written narrative + an auto-generated directory tree.

- **Narrative (hand-written)** explains *why / what* — purpose of each area and its key entry-point files. This is the part that gives onboarding and AI value, and it stays stable.
- **Auto-tree (generated)** shows the real directory structure, regenerated by a script so it never goes stale. It lives between explicit markers; the generator only ever rewrites that span.

**Generator language: Python (stdlib only).** Chosen over shell `find`/`tree` because the generator must (a) parse YAML frontmatter `title:` to annotate entries and (b) collapse `contents/` at the BU level conditionally — both awkward and brittle in shell. Precedent exists (`scripts/*.py`, e.g. `build_faq_seed_sql.py`). No third-party dependencies.

## Deliverables

1. `sitemap.md` (repo root) — narrative sections + an auto-tree span between markers.
2. `scripts/gen_sitemap.py` — regenerates the auto-tree span in place (idempotent), with a `--check` mode.
3. One line added to `CLAUDE.md` ("Where to look next") pointing at `sitemap.md`.

## `sitemap.md` structure

```
# Repository Sitemap
  Purpose: who reads this, how to use it, how to regenerate the tree.

## Top-level map            [hand]  table: top-level dir | one-line purpose | pointer to deeper docs
## backend/  — Go Fiber API [hand]  entry points: cmd/server/main.go, internal/api/chat_stream_flow.go,
                                    internal/services/*, config/{tuning,intents,path_rules,prompts}.yaml,
                                    migrations/
## frontend/ — Next.js      [hand]  routes: app/{categories,faq,chat,activity,admin}, lib/config.ts
## contents/ — markdown SoT [hand]  the 3 BUs (carmen, blueledgers, training_center) + each BU's top
                                    categories, with .md counts; NOT per-file
## scripts/ / docs/ / manual/ / root config
                            [hand]  ops scripts, specs+plans, runbooks, deploy (docker-compose,
                                    render.yaml, go.work, run_dev.sh)
## Directory tree (auto-generated)
                            [auto]  generated by scripts/gen_sitemap.py
   <!-- BEGIN AUTO-TREE -->
   ...tree...
   <!-- END AUTO-TREE -->
```

- `[hand]` sections are authored manually and explain purpose/meaning.
- `[auto]` is the only span the generator touches.
- A closing note states: edit narrative by hand freely, but never edit inside the markers; run `scripts/gen_sitemap.py` to refresh the tree.

## Generator: `scripts/gen_sitemap.py`

### Behavior

- Walk the directory tree from repo root, depth-bounded (code dirs ~2–3 levels).
- **Ignore:** `.git`, `node_modules`, `__pycache__`, `.next`, `dist`, `*.tsbuildinfo`, and `_images/` directories inside `contents/` (otherwise the tree explodes with image folders).
- **Collapse `contents/` at BU level:** show `contents/<bu>/` and its first-level categories only, each annotated with a `.md` count (e.g. `carmen/ (103 md)`). Do not descend into individual files.
- **Annotate frontmatter `title:`** for entry-point / `index.md` files where present (optional; append only when a title exists).
- Rewrite only the span between `<!-- BEGIN AUTO-TREE -->` and `<!-- END AUTO-TREE -->` in `sitemap.md`. Narrative outside the markers is untouched.
- If `sitemap.md` is missing, or the markers are absent/malformed, exit with a clear error — never overwrite blindly.
- Stdlib only (`os`/`pathlib`, `argparse`, `re`). No dependencies.

### Invocation

- `python3 scripts/gen_sitemap.py` — regenerate the tree span in place.
- `python3 scripts/gen_sitemap.py --check` — exit non-zero if the on-disk tree span differs from freshly generated output (no write). Intended for optional future CI gating; not wired up now.

### Configuration (in-script constants, no flags needed)

- `IGNORE` set (dirs/globs above).
- `MAX_DEPTH` for general code dirs.
- `CONTENTS_DIR` = `contents` with BU-level collapse + `.md` counting.
- Marker strings.

## Maintenance / integration

- Closing note in `sitemap.md` documents the hand-vs-auto split and the regenerate command.
- One line in `CLAUDE.md` "Where to look next" → `sitemap.md`.
- No git hook / CI this round (YAGNI). `--check` exists so CI gating is a one-line add later.

## Testing

- **Unit-ish (generator on a fixture):** run against a small temp fixture directory and assert:
  - (a) only the marker span is replaced;
  - (b) narrative text outside the markers is preserved verbatim;
  - (c) `--check` returns non-zero when the span is stale and zero when current;
  - (d) missing/malformed markers produce a clear error and no write.
- **Smoke test:** run on the real repo, inspect `git diff` to confirm the tree is sensible (BUs collapsed, ignores applied, no `_images` noise).

## Open questions / risks

- `--check` exact-match is sensitive to whitespace/ordering — the generator must produce deterministic output (sorted entries) so `--check` is stable.
- Frontmatter title annotation is best-effort; files without a YAML block are listed without a title (no error).
